大模型很强，但它并不是万能的。在实际项目中，我们经常会遇到几个问题：

- 大模型不知道企业内部私有知识
- 大模型的知识可能过时
- 大模型回答时可能出现幻觉
- 直接把大量文档塞进 prompt 成本高、效果也不稳定

比如我们想做一个企业知识库 问答 系统，用户问：

> 公司报销流程是什么？

如果这个流程只存在于公司内部文档里，通用大模型本身是不知道的。  
这时候就需要 RAG。

RAG，全称是 **Retrieval-Augmented Generation** ，中文一般叫 **检索增强生成** 。

它的核心思想是：

> 先从外部知识库中检索和问题相关的内容，再把这些内容作为上下文交给大模型，让大模型基于检索结果生成答案。

简单来说，RAG 不是让大模型“凭记忆回答”，而是让它“带着资料回答”。

## 1\. RAG 的整体架构设计

一个完整的 RAG 系统通常可以分成三个核心阶段：

```haskell
Indexing  ->  Retrieval  ->  Generation

索引构建      检索召回        生成答案
```

如果再细一点，可以拆成：

```haskell
数据获取

-> 文档预处理

-> 文档分块

-> 向量化

-> 存入向量数据库

-> 用户提问

-> 查询向量化

-> 相似度检索

-> 检索结果重排

-> 构建 Prompt

-> 调用大模型生成答案

-> 答案后处理与评估
```

整体流程可以理解为：

```
知识文档

  ↓

清洗、分块、向量化

  ↓

向量数据库

  ↓

用户问题

  ↓

检索相关知识片段

  ↓

拼接 Prompt

  ↓

LLM 生成答案
```

RAG 中有两个关键链路：

第一条是 **离线索引链路** ，也就是把知识库内容提前处理好，存入向量数据库。

第二条是 **在线问答链路** ，也就是用户提问时，实时检索相关知识，再调用大模型生成答案。

## 2\. Overview：RAG 是什么？

RAG 的核心由两个部分组成：

- **Retrieval** ：检索，从知识库中找到相关内容
- **Generation** ：生成，基于检索内容生成答案

传统大模型问答一般是：

```haskell
用户问题 -> 大模型 -> 答案
```

RAG 问答则变成：

```haskell
用户问题 -> 检索知识库 -> 获取相关文档片段 -> 大模型 -> 答案
```

这样做的好处是：

1. **减少幻觉**

大模型不再完全依赖自身参数中的知识，而是基于外部文档回答。

1. **支持私有知识**

企业文档、产品手册、业务规则、内部 FAQ 都可以接入 RAG。

1. **知识更新更方便**

如果知识变了，不需要重新训练模型，只需要更新知识库索引。

1. **回答更可控**

可以要求模型只基于检索内容回答，不知道就说不知道。

RAG 常见应用场景包括：

- 企业知识库问答
- 智能客服
- 文档助手
- 法律、医疗、金融资料问答
- 论文问答
- 产品手册问答
- 代码仓库问答

一句话总结：

> RAG 是一种让大模型结合外部知识库回答问题的技术方案，核心目标是提升答案准确性、时效性和可控性。

## 3\. Indexing：索引构建

Indexing 是 RAG 的第一阶段，目标是把原始知识文档处理成可以被检索的形式。

这一阶段通常是离线完成的。

### 3.1 数据获取 Data Ingestion

RAG 的知识来源可以有很多种：

- PDF 文档
- Word 文档
- Markdown 文档
- HTML 网页
- 数据库内容
- FAQ 问答对
- 产品手册
- 接口文档
- 企业内部知识库

数据获取阶段的任务是把这些不同来源的数据统一加载进系统。

比如：

```haskell
PDF -> 提取文本

网页 -> 抓取正文

数据库 -> 查询业务数据

Markdown -> 解析标题和正文
```

实际项目中，数据源越复杂，数据接入成本越高。尤其是 PDF、扫描件、表格、图片混排文档，解析质量会直接影响后续检索效果。

---

### 3.2 文档预处理 Preprocessing

拿到原始文档后，不能直接丢进向量 数据库 ，一般要先做预处理。

常见预处理包括：

- 去除空行、乱码、页眉页脚
- 清理无意义字符
- 去除重复内容
- 保留标题层级
- 提取文档来源、作者、时间等元信息
- 统一编码格式
- 对表格、代码块做特殊处理

比如一篇文档中可能有很多重复页脚：

```cobol
版权所有 xxx 公司 第 1 页

版权所有 xxx 公司 第 2 页
```

这些内容对问答没有帮助，反而会污染检索结果。

预处理阶段还要尽量保留文档结构，比如：

```
一级标题

二级标题

正文

列表

表格
```

因为这些结构信息对后续分块和答案生成很有帮助。

---

### 3.3 文档分块 Chunking

文档分块是 RAG 中非常关键的一步。

为什么要分块？

因为原始文档通常很长，不能直接整体向量化，也不能每次全部塞进 prompt。

如果一整篇文档都作为一个 chunk，会导致：

- 向量表达不精确
- 检索结果太粗
- prompt 太长
- 无关内容太多

如果 chunk 太小，又会导致：

- 上下文不完整
- 语义被切断
- 模型拿不到足够信息

所以分块的目标是：

> 每个 chunk 尽量保持语义完整，同时长度适中，方便检索和生成。

常见分块策略有几种。

**固定长度分块**

按固定字符数或 token 数切分。

```cobol
每 500 个 token 切一块
```

优点是实现简单。  
缺点是可能把一个完整段落切断。

**按标题分块**

根据 Markdown 标题、Word 标题、HTML 标题切分。

```cobol
第 1 章

  1.1 小节

  1.2 小节
```

优点是语义更完整。  
缺点是不同文档结构不统一时处理复杂。

**滑动窗口分块**

相邻 chunk 之间保留一部分重叠内容。

比如：

```cobol
chunk size = 500 tokens

overlap = 100 tokens
```

这样可以避免关键信息刚好被切断。

在实际项目中，比较常见的是：

```
按标题或段落优先切分

如果块太长，再按 token 长度切分

相邻块之间保留一定 overlap
```

---

### 3.4 向量化 Embedding

分块完成后，需要把每个文本 chunk 转成向量。

这个过程叫 embedding。

Embedding 的作用是把文本映射到一个高维向量空间中。语义相近的文本，在向量空间中的距离也更近。

比如：

```
“如何申请报销？”

“报销流程是什么？”
```

这两个问题字面不完全一样，但语义接近，向量距离也应该比较近。

向量化后，一个 chunk 可能变成类似这样的数组：

```cobol
[0.013, -0.241, 0.087, ..., 0.552]
```

这个向量后续会存入向量数据库，用于相似度检索。

常见 embedding 模型有：

- OpenAI Embedding
- BGE
- text2vec
- GTE
- E5
- 各类国产 embedding 模型

选择 embedding 模型时，要关注：

- 中文效果
- 向量维度
- 推理速度
- 成本
- 是否支持私有化部署
- 和业务领域的匹配程度

---

### 3.5 存储到向量数据库

向量化完成后，需要把数据存储到向量数据库中。

通常存储三类信息：

```
chunk 文本

chunk 向量

metadata 元信息
```

metadata 很重要，比如：

```cobol
{

  "docId": "refund_policy_001",

  "title": "报销制度",

  "source": "公司制度文档",

  "page": 3,

  "section": "差旅报销"

}
```

有了 metadata，后续可以做到：

- 按文档过滤
- 按时间过滤
- 按权限过滤
- 返回答案来源
- 做引用展示

常见向量数据库包括：

- FAISS
- Milvus
- Chroma
- Weaviate
- pgvector
- Elasticsearch 向量检索

向量数据库的核心能力是：

> 根据 query 向量快速找到最相似的 topK 个文档片段。

---

## 4\. Retrieval：检索

Retrieval 是 RAG 的第二阶段。

当用户输入问题后，系统需要从向量数据库中找出和问题最相关的知识片段。

### 4.1 查询处理

用户的问题通常不能直接拿去检索，可能需要做一些处理。

常见处理包括：

- 去除无意义字符
- 补全上下文
- 多轮对话问题改写
- 关键词提取
- 查询重写
- 语言规范化

比如多轮对话中：

```typescript
用户：Redis 有哪些数据结构？

助手：String、Hash、List、Set、ZSet...

用户：它适合做排行榜吗？
```

第二个问题里的“它”指的是 Redis。  
如果直接检索“它适合做排行榜吗”，效果会比较差。

所以可以把问题改写成：

```
Redis 适合做排行榜吗？
```

查询处理的目标是：

> 让用户问题变得更清晰、更适合检索。

---

### 4.2 向量检索

查询处理后，需要把用户问题也转成向量。

流程是：

```haskell
用户问题

-> embedding 模型

-> query vector

-> 向量数据库相似度检索

-> 返回 topK chunk
```

常见相似度计算方式有：

- cosine similarity
- dot product
- euclidean distance

比如用户问：

```
公司差旅报销标准是什么？
```

向量数据库可能返回：

```cobol
chunk1：差旅报销标准

chunk2：交通费用报销规则

chunk3：住宿费用报销规则

chunk4：发票提交要求
```

这些 chunk 会作为后续生成阶段的上下文。

---

### 4.3 检索策略

基础 RAG 只做向量 topK 检索，但真实项目中通常不够。

因为向量检索可能出现：

- 召回内容不完整
- 召回内容语义相关但不能回答问题
- 关键词精确匹配能力不足
- topK 中混入无关 chunk

所以常见优化策略包括以下几类。

**topK 检索**

返回相似度最高的 K 个 chunk。

```cobol
topK = 3 / 5 / 10
```

K 太小可能漏掉关键信息。  
K 太大可能引入噪声，并增加 prompt 长度。

**score threshold**

设置相似度阈值。

```cobol
相似度低于 0.7 的结果丢弃
```

这样可以避免把明显不相关的内容喂给大模型。

**Hybrid Search**

混合检索，也就是：

```cobol
向量检索 + 关键词检索
```

向量检索擅长语义相似。  
关键词检索擅长精确匹配。

比如产品型号、错误码、订单号、接口名，关键词检索往往更准确。

**Rerank 重排**

先召回较多候选结果，比如 top20。  
再用 rerank 模型重新排序，选出最相关的 top5。

流程：

```cobol
向量召回 top20

-> rerank

-> 取 top5

-> 交给 LLM
```

Rerank 通常能明显提升检索质量。

**Metadata 过滤**

根据元信息过滤文档。

比如：

```cobol
只检索用户有权限访问的文档

只检索 2024 年之后的制度

只检索某个产品线的知识库
```

这在企业知识库中非常重要。

---

### 4.4 检索效果评估指标

RAG 效果好不好，很大程度取决于检索质量。

如果检索阶段找错了内容，生成阶段再强也很难答对。

常见检索指标有：

**Recall@K**

表示正确答案相关文档是否出现在前 K 个结果中。

比如：

```cobol
Recall@5 = 0.8
```

说明 80% 的问题，相关文档能在前 5 个结果里找到。

**MRR**

Mean Reciprocal Rank，平均倒数排名。

如果正确结果排得越靠前，MRR 越高。

**NDCG**

用于评价排序质量。  
不仅看有没有召回，还看相关结果排得是否靠前。

**命中率**

可以简单理解为：

```
检索结果中是否包含能回答问题的 chunk
```

实际项目中，也可以人工标注一些问题集，然后检查：

- 检索结果是否相关
- 是否覆盖答案所需信息
- 是否有无关干扰内容
- top1、top3、top5 命中率如何

## 5\. Generation：生成

Generation 是 RAG 的第三阶段。

这个阶段会把用户问题和检索到的知识片段组织成 prompt，交给大模型生成最终答案。

### 5.1 提示词构建 Prompt Engineering

Prompt 构建非常重要。

一个基础 RAG prompt 通常包含：

- 角色设定
- 回答规则
- 检索上下文
- 用户问题
- 输出格式

比如：

```
你是一个知识库问答助手。

请只根据给定的知识库内容回答用户问题。

如果知识库中没有答案，请回答“根据当前知识库无法确定”。

不要编造不存在的信息。

 

知识库内容：

{context}

 

用户问题：

{question}

 

请给出答案：
```

这里最重要的是约束模型：

```
只根据检索内容回答

不知道就说不知道

不要编造
```

这样可以减少幻觉。

如果需要引用来源，也可以要求模型输出：

```
答案

参考来源
```

比如：

```cobol
答案：差旅报销需要在出差结束后 7 个工作日内提交。

来源：《差旅报销制度》第 3 页
```

---

### 5.2 调用 LLM 生成

构建好 prompt 后，就可以调用大模型生成答案。

流程是：

```
用户问题

+ 检索上下文

+ 回答规则

-> LLM

-> 最终答案
```

在实际项目中，还要注意上下文长度。

如果检索结果太多，prompt 会过长，带来几个问题：

- 成本增加
- 响应变慢
- 超过模型上下文限制
- 无关内容干扰回答

所以一般会控制：

```
topK 数量

单个 chunk 长度

总 context token 数
```

必要时还可以先对检索结果做摘要，再交给模型生成。

---

### 5.3 答案后处理

模型生成答案后，通常还要做一些后处理。

常见后处理包括：

- 去除多余格式
- Markdown 格式化
- 敏感词过滤
- 添加引用来源
- 检查答案是否为空
- 判断是否命中兜底回答
- 统一返回结构

比如最终返回给前端：

```cobol
{

  "answer": "差旅报销需要在出差结束后 7 个工作日内提交。",

  "sources": [

    {

      "title": "差旅报销制度",

      "page": 3

    }

  ]

}
```

如果检索结果为空，可以直接返回：

```
当前知识库中没有找到相关内容。
```

不一定要继续调用大模型。

### 5.4 完整 RAG 生成流程伪代码

下面是一个简化版 RAG 流程：

```java
public String ask(String question) {

    // 1. 查询改写

    String rewrittenQuestion = queryRewrite(question);

 

    // 2. 问题向量化

    List<Double> queryVector = embeddingModel.embed(rewrittenQuestion);

 

    // 3. 向量检索

    List<Chunk> chunks = vectorStore.search(queryVector, 5);

 

    // 4. 过滤低分结果

    List<Chunk> filteredChunks = chunks.stream()

            .filter(chunk -> chunk.getScore() >= 0.7)

            .toList();

 

    if (filteredChunks.isEmpty()) {

        return "当前知识库中没有找到相关内容。";

    }

 

    // 5. 构建上下文

    String context = buildContext(filteredChunks);

 

    // 6. 构建 Prompt

    String prompt = """

            你是一个知识库问答助手。

            请只根据以下知识库内容回答问题。

            如果无法从知识库中得到答案，请说明无法确定。

            知识库内容：

            %s

            用户问题：

            %s

            """.formatted(context, question);

 

    // 7. 调用大模型

    String answer = llm.chat(prompt);

 

    // 8. 后处理

    return postProcess(answer);

}
java运行
```

真实项目中会更复杂，比如：

- 多轮对话记忆
- 权限过滤
- hybrid search
- rerank
- 流式输出
- 引用来源
- 答案质量评估

但核心流程基本就是这几步。

---

### 5.5 生成效果评估

RAG 不仅要评估检索效果，也要评估最终生成效果。

常见生成评估维度包括：

**Answer Relevance**

答案是否真正回答了用户问题。

**Faithfulness**

答案是否忠于检索内容。  
如果答案里出现了检索内容中没有的信息，就可能是幻觉。

**Completeness**

答案是否完整覆盖问题所需信息。

**Conciseness**

答案是否简洁，没有大量无关内容。

**Citation Accuracy**

如果系统提供引用来源，需要判断引用是否准确。

实际项目中，可以准备一批测试问题：

```
问题

标准答案

相关文档

模型答案
```

然后从检索和生成两个角度评估：

```
是否检索到正确文档

答案是否正确

是否出现幻觉

引用是否准确
```

## 6\. RAG 常见问题与优化方向

### 6.1 分块不合理

如果 chunk 太大，检索结果会包含太多无关内容。  
如果 chunk 太小，答案上下文可能不完整。

优化方式：

- 按标题和段落切分
- 控制 chunk token 数
- 设置 overlap
- 保留章节标题作为 metadata

---

### 6.2 检索不准确

检索不准确通常是 RAG 效果差的主要原因。

优化方式：

- 更换更适合中文或业务领域的 embedding 模型
- 使用 hybrid search
- 加 rerank
- 优化 query rewrite
- 增加 metadata 过滤
- 调整 topK 和 score threshold

---

### 6.3 上下文太长

检索结果太多会导致 prompt 过长。

优化方式：

- 限制 topK
- 设置最大 context token
- 对 chunk 做重排
- 对长内容先摘要
- 去除重复 chunk

---

### 6.4 大模型幻觉

即使使用 RAG，大模型仍然可能幻觉。

优化方式：

- prompt 中明确要求只基于上下文回答
- 检索不到内容时不调用模型
- 要求输出引用来源
- 使用答案校验模型判断是否忠于原文
- 对高风险场景增加人工审核

---

### 6.5 权限问题

企业知识库中，不同用户可能只能访问不同文档。

如果不做权限控制，可能出现越权回答。

优化方式：

- 文档写入时存储权限 metadata
- 检索时根据用户权限过滤
- 不把无权限 chunk 放进 prompt
- 答案引用也要经过权限校验

---

## 7\. 总结

RAG 的核心不是让大模型记住所有知识，而是让大模型在回答问题前，先从外部知识库中检索相关内容，再基于这些内容生成答案。

真正影响 RAG 效果的，往往不是单个模型能力，而是整条链路的设计：

- 文档清洗是否干净
- chunk 是否合理
- embedding 是否适合业务
- 检索策略是否准确
- prompt 是否约束清楚
- 是否有评估和优化机制
