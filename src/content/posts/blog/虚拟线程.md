# 1.  什么是虚拟线程 ？

虚拟线程是JDK21引入的一个新特性，它是运行在平台线程上的极其轻量级的线程，让你可以轻松开启成千上万个线程且不用担心系统崩溃。

# 2. 虚拟线程为什么要出现 ？

在虚拟线程出现之前，Java的线程是平台线程，每一个Java线程对应着一个操作系统内核线程，内核线程很“重”。创建一个线程需要分配大约 1MB 的栈内存，而且上下文切换（CPU 从一个线程换到另一个线程）需要经过昂贵的系统调用。高并发场景下，大量的请求io会导致大量线程阻塞等待，最终导致cpu资源耗尽。

**具体实例：**

对比在**固定线程池（平台线程）和虚拟线程**下，服务器的表现。

###### 1. 传统方案：平台线程池

```
// 模拟一个拥有 100 个线程的线程池
ExecutorService executor = Executors.newFixedThreadPool(100);

long start = System.currentTimeMillis();

// 提交 1,000 个任务，每个任务模拟 1 秒的阻塞调用
for (int i = 0; i < 1000; i++) {
    executor.submit(() -> {
        try {
            Thread.sleep(1000); // 模拟 I/O 阻塞：等 API 返回
        } catch (InterruptedException e) { }
    });
}

executor.shutdown();
executor.awaitTermination(1, TimeUnit.HOURS);
System.out.println("总耗时: " + (System.currentTimeMillis() - start) + "ms");
```
- **发生过程：**
    
    1. 前 100 个请求进来，占满了 100 个线程。
        
    2. 这 100 个线程全都在 `Thread.sleep`（阻塞），**虽然 CPU 闲着，但线程被占死了**。
        
    3. 剩下的 900 个请求只能在队列里排队。
        
    4. 每过 1 秒，才能处理完 100 个请求。
        
- **最终结果：** 处理完 1,000 个请求需要约 **10 秒**。
    
- **痛点：** 哪怕 CPU 利用率不到 1%，你的服务器也快不起来，因为“坑位”（线程）有限。

###### 2. 虚拟线程

```
// 虚拟线程执行器：随用随开
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    long start = System.currentTimeMillis();

    for (int i = 0; i < 1000; i++) {
        executor.submit(() -> {
            try {
                Thread.sleep(1000); // 这里的阻塞会触发“卸载”
            } catch (InterruptedException e) { }
        });
    }
    // try-with-resources 会等待所有虚拟线程执行完毕
}
System.out.println("总耗时: " + (System.currentTimeMillis() - start) + "ms");
```
- **发生过程：**
    
    1. 1,000 个虚拟线程瞬间被创建（它们只占用极少内存）。
        
    2. 当第一个虚拟线程执行到 `Thread.sleep` 时，JVM 发现它要阻塞了。
        
    3. **核心魔法：** JVM 立即把这个虚拟线程从底层的内核线程（Carrier Thread）上“卸载”下来，让内核线程去处理第二个、第三个……第 1,000 个虚拟线程。
        
    4. 这 1,000 个虚拟线程几乎**同时**进入了等待状态，且没有霸占昂贵的内核线程。
        
- **最终结果：** 处理完 1,000 个请求只需要约 **1 秒多一点**。

# 总结

虚拟线程最大的好处不是让单次任务变快（它甚至比内核线程慢一点点点），而是极大地提高了系统的并发能力（吞吐量）。它让你可以用阻塞式的简单代码，跑出非阻塞式的超高性能。
